# Chili (claude branch) — 0.7.5 release notes

**Date:** 2026-04-26
**Branch:** `claude` (local fork)
**Workspace version:** 0.7.4 (Rust). **Python wheel version:** `chili-pie 0.7.5`.
**Audience:** downstream consumers — mdata, nxcar, or any other Python project that imports `chili`.

This release rewrites the chili-py FFI to remove the Arrow IPC bytes round-trip on the eval/write paths. Function signatures changed at the boundary; user-facing semantics are otherwise the same.

---

## TL;DR — what you have to change in mdata / nxcar

If your code looks like this:

```python
ipc = engine.eval("select from ohlcv_1d where date=2024.01.02")
table = pyarrow.ipc.open_stream(io.BytesIO(ipc)).read_all()
df = polars.from_arrow(table)
```

Replace it with:

```python
df = engine.eval("select from ohlcv_1d where date=2024.01.02")  # already a polars.DataFrame
```

If your code looks like this:

```python
buf = io.BytesIO()
df.write_ipc_stream(buf)
engine.wpar(buf.getvalue(), "/hdb", "ohlcv_1d", "2024.01.02")
```

Replace it with:

```python
engine.wpar(df, "/hdb", "ohlcv_1d", "2024.01.02")
```

That's it. No other API surface changed.

## What changed at the FFI boundary

| Method | Before (0.7.4) | After (0.7.5) |
|---|---|---|
| `engine.eval(q)` | returns `bytes` (Arrow IPC stream); Python wrapper decodes via `pa.ipc.open_stream + pl.from_arrow` | returns `polars.DataFrame` directly via `pyo3_polars::PyDataFrame` (zero-copy wrapper) |
| `engine.wpar(df, ...)` | Python wrapper called `df.write_ipc_stream(buf)` then passed `buf.getvalue()` to FFI; FFI took `&[u8]` and reparsed via `IpcStreamReader` | FFI takes `PyDataFrame` directly. No serialize/deserialize on the input path. |
| `engine.overwrite_partition(df, ...)` | same as wpar | same as wpar |
| `engine.tick_upd(table, df)` | Python wrapper called `df.write_ipc_stream(buf)` then `engine.publish(table, buf.getvalue())` | Rust-side IPC serialize with GIL released, then internal publish. Subscriber callback contract unchanged (still receives bytes). |
| `engine.publish(topic, ipc_bytes)` | bytes in | bytes in (unchanged — generic byte payload publishing is still useful) |
| `engine.subscribe(topics, callback)` | callback receives `(topic, seq, ipc_bytes)` | unchanged |

## Latency win

Microbenchmark: small `eval()` (3-row result) — **~9 µs/call** end-to-end, single-threaded, on M-series Apple silicon. Previously the IPC round-trip alone cost ~50–200 µs on small results and ~5–10 ms each side on multi-MB results. The win scales with result size: zero-copy is constant-time at the boundary.

GIL release behavior unchanged: `eval()`, `wpar()`, `overwrite_partition()`, `tick_upd()`, `query_plan()`, and `load()` all still wrap their Rust bodies in `Python::detach` (formerly `allow_threads`). The 8-thread concurrent throughput number from the optimization sweep (3168 q/s, 6.10× single-thread) holds.

## Why this changed

Upstream `purple-chili/chili` author committed his own Python bindings (`bf9fa14`) using the canonical `pyo3_polars::PyDataFrame` zero-copy wrapper. Our original 0.7.4 design used IPC bytes as a workaround for a non-issue (Phase-2 `SpicyObj: Send` concerns that turned out to be untrue with `pyo3-polars`). The author's path is strictly faster. We adopted it.

Lessons baked into memory: reach for the canonical ecosystem bridge (`pyo3-polars::PyDataFrame`) before designing a workaround.

## What was NOT changed (intentional)

- **Package name stays `chili-pie`.** Upstream renamed his to `chili-sauce`. mdata and nxcar import `chili-pie`; renaming would force a coordinated downstream change. Will revisit when `chili-sauce` is actually published to PyPI.
- **The `Engine` class API is unchanged.** Same methods (`load`, `eval`, `wpar`, `overwrite_partition`, `publish`, `subscribe`, `tick_upd`, `broker_eod`, `set_column_scale`, `clear_column_scales`, `query_plan`, `stats`, `is_loaded`, `table_count`, `reload`, `unload`, `close`) and same arguments — only the bytes-vs-DataFrame I/O changed.
- **All structured exceptions are preserved**: `ChiliError`, `PepperParseError`, `PepperEvalError`, `PartitionError`, `TypeMismatchError`, `NameError`, `SerializationError`. Phase 13 contract unchanged.
- **Fork-safety, GIL release, broker pubsub, Int64 column-scale dequant, query_plan** — all retained.

## Workspace dependency changes

- pyo3: `0.22` → `0.27`
- pyo3-polars: not used → `0.26.0`
- chrono: per-crate `"0.4"` → workspace dep
- pyarrow: removed from `chili-pie` runtime deps
- min Python: 3.7 → 3.10

If you maintain a Rust crate that depends on `chili-core` / `chili-op`, the changes above are FFI-only — your Rust code is unaffected.

## Test coverage

- **35 baseline pytest tests** (broker, partition_filter, extras) — all pass on the new FFI.
- **9 new regression tests** (`tests/test_direct_dataframe.py`) pinning:
  - `eval()` returns `pl.DataFrame`, not bytes
  - dtype preserved across the FFI (no Arrow round-trip can lose category/Int32)
  - `wpar()` rejects bytes input (regression guard for old API)
  - `overwrite_partition()` accepts DataFrame directly
  - `tick_upd()` publishes from DataFrame, subscriber receives valid IPC bytes
  - all 7 structured exception classes exported and import-safe post-pyo3-bump
  - `PartitionError` still raised for missing date predicate
